# 設計書 — CMS からの移行(静的化+自営スタック)

動的CMS から、静的配信+自営スタック(共同編集・問い合わせ・業務)へ移行する
ための設計書。サーバー運用の担当者が構築・運用・引き継ぎに用いる。移行と
生成のツール一式は cms-migration-kit(CMS移行キット。本リポジトリ)を使う。

## 0. 設計の考え方(CMS の「仮」を本格にする)

CMS は、本体で扱えない業務を仮の手段へ押し出す。依頼試験・設備予約・研修は
担当PCの**その場しのぎの Excel**・紙・メールで処理され、フォームは後付け
プラグイン、検索や解析は外部サービス頼み、内容編集は「ログインして直す」
だけで承認も版管理も薄い。結果として業務が分断・属人化し、CMS 自身の機能も
後付けの仮になりがち。

本構成は、その仮を一つずつ本格の形に引き上げる:

- 内容 → git で版管理・共同編集・承認(本格の編集プロセス)
- 申込・問い合わせ → 様式(神Excel)を唯一の定義とする機械可読なデータ経路。
  **これは仮ではなく本格の部品**——VBA(文書内に特権実行コードが同居)を、
  外部Python(処理)+内部JavaScript(サンドボックスの補助)に分離し、
  「開く=実行」という攻撃経路を設計から無くしている(§5)
- 業務(試験・予約・研修) → 認証・DB を備えたアプリ(その場しのぎ Excel・紙の置換)
- 認証 → アプリ層で職員・会員を role 分離(PocketBase)。編集層は Forgejo の内蔵認証
- メール → 自営で受付・振り分け(代表フォーム一本の置換)

狙いはコスト削減以上に、**仮の回避策を無くし、所有・統合・監査できる形に
すること**。ただし本格化は運用の引き受けを伴う(§1 の運用形態)。全業務を
無条件に本格化するのではなく、業務ごとに要否で水準を選ぶ(単独・小規模の
部分は静的化までで十分な場合がある)。

### 三つの閉じた世界からの開放(Microsoft 365・業務システム・CMS)

機関の情報環境は、三つの閉じた世界に分かれてきた——**Microsoft 365**
(文書・メール・会議。席数課金・外部クラウド)、**業務システム**(基幹・
業務パッケージ。独自形式・独自API・ベンダー保守契約)、**CMS**(公開Web。
ライセンス・保守・脆弱性対応)。それぞれが独自形式と契約で閉じ、データは
外に出しにくく、世界の境界のたびに転記と受け渡しが生じる(冒頭の
「分断・属人化」の正体)。

本構成は、この三つを**一つの開いた土台**(認証・SQL・OnlyOffice・メール・
git・静的公開)で置き換える。公開Web側から見れば「CMS移行キット」、業務・
文書・メール側から見れば「Microsoft 365 移行キット」——同じ道具の顔が違う
だけである。業務システムは同じ土台の上のモジュールになる(§13)。
Microsoft 365 との対応:

| Microsoft 365 | 本構成(自営) |
|---|---|
| Exchange(メール・予定表) | Stalwart(SMTP/IMAP/JMAP・CalDAV)+ Thunderbird |
| Word/Excel・Office 文書 | OnlyOffice |
| OneDrive / SharePoint(文書共有) | 自作文書ホスト + OnlyOffice Docs(§2) |
| Teams(会議・ウェビナー) | Jitsi / BigBlueButton |
| Bookings(予約) | Cal.com |
| Copilot(AI) | ローカルAI(Command A+ / North Mini Code) |
| VBA / Power Automate(自動化) | 外部Python + 内部JavaScript |
| SharePoint / CMS(公開) | 静的HTML(Cloudflare)+ Forgejo(git) |
| 席数課金のクラウド | 自営 OSS(継続利用料なし) |

部品の選定は自立編スタック(https://aiseed.dev/ai-native-ways/software/)に
よる。会議・ウェビナー・予約は必要になった機関で足す(本キットの必須部品では
ない)。Teams のチャット相当は本キットの範囲外(必要時に選定する)。

いずれも「ベンダーの檻から手元へ」——所有・限界費用ゼロ・データを建物の中に、
という同じ原則の適用先が違うだけである。

### 閉じた世界をつくらない(標準でつなぐ)

目的は、三つの閉じた世界を**別の閉じた世界(自前スタックへの新たな
ロックイン)に置き換えることではない**。部品は独自形式・独自 API でなく、
少数の公開標準で疎結合に繋ぐ:

- データ = **Markdown / YAML / JSON / Parquet**(自己記述・スキーマ付き。
  位置がずれると黙って壊れる CSV は境界に使わない)
- 問い合わせ・蓄積 = **SQL**(実装は PostgreSQL・SQLite。製品ではなく
  SQL にコミットする)
- 処理・接着 = **Python**、体裁 = テンプレート(Jinja2)

自己記述で・AI が扱えて・多くの道具が読める形式だから、部品は差し替え
可能で、データはツールより長生きし、**この仕組み自体からも出られる**。
檻から出ることと、出た先を檻にしないこと——その両方で「開放」である。

### Claude の役割(開放を現実にする)

OSS はもともと無料だった。三つの世界に機関を閉じ込めてきたのは、OSS を
組み立て・維持する労力と専門性である。**Claude がその労力を担うことで、
自営=開放が初めて現実的になった**。

- 担うもの: 構築(設定・接着コード)、移行(内容の変換・下書き)、保守
  (調整・改修のコード化)。人に残るのは確認・判断・内容の最終責任で、
  手作業に比べ時間は桁違いに小さい(ただしゼロではない)
- 席は**運用責任者の1席**で足りる。ほかの職員は無償の OSS(Forgejo・Zed・
  OnlyOffice)と共有のローカルAI(North Mini Code・Command A+)で完結する。
  M365 の「席数 × 全員」とは逆で、**費用が利用者数に比例しない**
- 内部秘・個人情報・現場の局所はローカルAI と分担する(使い分けは §7)
- **Claude 自体も檻にしない**: 使うのは開発・移行・保守のときだけで、出力
  (Python・JS・Markdown・YAML)は人が確認して git に凍結し、ランタイム・
  処理経路には残さない。成果物はすべて公開標準の形式なので、Claude を
  やめても仕組みはそのまま動き、別の道具でも保守を続けられる

正直な一線: これは「今こそ現実的になった」であって楽勝ではない。正しさの
検証・段階移行・最終責任(機関)は残る。AI は労力を桁違いに下げるが、
判断をゼロにはしない。

### 最大の利点: CMS と Microsoft 365 を一つの移行で

普通は別プロジェクトになる「CMS 脱却」「Microsoft 365 脱却」「業務システムの
置き換え」が、**土台を共有するため一つの移行に畳める**。サーバースタック
(PostgreSQL・メール・OnlyOffice・git・認証)・ローカルAI が、Web 側と業務側の
両方を担う。OnlyOffice は申込様式の編集と日常の Office 文書を、メールは
問い合わせ受付と職員メールを、同じ基盤で兼ねる。結果、二重投資・二重運用・
両者の継ぎ目が生じない。

「一気に」の意味は**基盤を一度に作る**こと。切替は段階でよい(公開分→
問い合わせ様式→文書・メール→業務モジュール)。基盤が共有なので、段階を
踏んでも二重にならない。

### 一つになることで業務が流れる

別々のシステム(CMS・M365・担当PCの Excel)は、境界ごとに手作業の転記と
受け渡しを生む。告知は CMS、申込はフォーム、名簿は Excel、報告は Word、
連絡は Exchange——データが繋がらず、境界のたびに入れ直す(§0 冒頭の
「分断・属人化」の正体)。一つの土台(共通の認証・データ・道具)に載ると、
この境界が消えて業務が一本の流れになる。

- 告知 → 申込 → 処理 → 名簿 → 実績 → 連絡 が一つのデータの流れになる
  (再入力なし。申込様式がそのまま名簿・台帳・受領メールへ流れる)
- 認証が一つなので記録が繋がる(同じ事業者の試験・設備・研修の履歴が一望)
- 文書・データ・メールが同じ基盤にあり、Web→様式→DB→文書→メールが
  継ぎ目なく動く

コスト・所有の先にある本質的な価値は、この**業務が流れる(分断・属人化・
再入力が消える)**ことである。

### そのうえ、ソフトは無料

本格化・業務の流れ・一つの移行——それを支える構成部品が、すべて
OSS / オープンウェイトである。**ソフトのライセンス料・席数課金・サブスクは
ゼロ**(a-blog 等の CMS ライセンスも M365 の席数課金も消える)。

| 部品 | ライセンス |
|---|---|
| Cloudflare Pages | 無料枠(公開静的で足りる) |
| Forgejo / OnlyOffice | AGPL |
| Stalwart / PocketBase / PostgreSQL | OSS |
| North Mini Code(Apache)/ Command A+ | オープンウェイト |
| Python / Jinja2 / git / Zed | OSS/無償 |

無料なのは**ソフト**である。残る費用は、ハードウェア(サーバー類は多くが既存
流用。ただしローカルAI は別で、North Mini Code は GPU 1枚で動くが、Command A+ は
大型モデルのため相応の GPU メモリへの一回性の投資が要る)・
電気代・構築と移行の一回性の手間・運用の人手・任意の運用支援。いずれも
利用者数に比例して恒久的に増える種類のものではない(限界費用ゼロに近い)。

そのうえ、**構築と移行の手間の大半は AI が担い、要るのは運用責任者の
1席だけ**である(上記「Claude の役割」)。結果、恒久的に増える費用はほぼ
残らない。

**構築も速い。** 部品がほぼ OSS なので、調達・ライセンス交渉・ベンダーの
見積待ち・契約というリードタイムがない(プロプライエタリなら数ヶ月かかる
ところ)。部品は標準的でコンテナ化されており、設定・接着コードは AI が書く。
——ただし速いのは基盤の立ち上げで、時間がかかるのは内容の変換と検証
(メール到達性・認証・様式の動作確認)。そこは段階的に、AI で加速しながら
進める(§10)。

## 1. 目的・範囲

- 公開サイトを静的HTML化し、Cloudflare Pages で配信する
- 内容(ソース)を git 上のマークダウンとデータで管理する
- 問い合わせフォーム(唯一の動的機能)をメール受付に置き換える
- 対象は公開情報の提供サイト。個人情報・認証を要する処理は機関内に閉じる
- **二層構成**とし、業務システム・決済・会員機能を載せられる土台を最初から持つ:
  - **公開層**(静的・匿名・個人情報なし。Cloudflare Pages)
  - **アプリ層**(機関内。認証・DB・メールを共有し、その上にモジュールを載せる)
- **認証(ID基盤)はアプリ層の必須コンポーネント**。業務・会員・決済の
  いずれも認証を前提とする(§13)
- 最小構成(案内サイト+メール問い合わせのみ)ではアプリ層は不要。
  モジュールを持つ時にアプリ層を立てる。使わない柔軟性を先に作り込むのではなく、
  モジュールが載る場所と認証土台を設計として定めておく、という趣旨

### 運用形態で構成が決まる(最重要の分岐)

構成の大きさは規模ではなく**運用形態=共同編集の有無**で決まる。

- **1人運用(単独更新)**: 静的サイト化だけで足りる。`.md` を直接編集 →
  build → 配信。Forgejo・OnlyOffice Docs・アプリ層は不要
- **共同編集(複数人)**: 「誰が編集し・誰が承認し・履歴をどう突き合わせるか」が
  必要になる——これは CMS が担っていた機能そのもの。よって
  Forgejo(git 共同編集。認証・権限は Forgejo 内蔵)+ Zed(+ OnlyOffice Docs)が
  要る。業務・会員・決済を持てばアプリ層も加わる。**結局スタック全体が必要になる**

多くの機関では内容を複数人で編集する。CMS の核はまさにこの「複数人編集」で
あり、置き換えで**最初に据えるのは共同編集の基盤=Forgejo(git)+Zed**である。
内容はここに載せて編集し、静的生成・配信・問い合わせ
様式・アプリ層モジュールはその上に順次足す。1人運用に限れば基盤を省き
静的化だけでよい(小規模の例外)。段階的に上げられ、土台は変えず上に積む。

## 2. 全体構成

CMS が一体で担っていた機能を、役割ごとに自営の道具へ分解する。公開層だけが
外(Cloudflare)に出て、編集・運用とアプリは機関内で自営する。すべて自営
OSS で、継続ライセンス料はない。

| CMS が担っていた機能 | 置き換え |
|---|---|
| 公開ページの配信 | 静的HTML + Cloudflare Pages(公開層) |
| 複数人での内容編集・版管理 | Forgejo(git 共同編集。認証内蔵)+ Zed(編集) |
| 文書・様式の編集 | OnlyOffice Docs(編集エンジン)+ 自作文書ホスト(ブラウザ編集) |
| 問い合わせフォーム処理 | 申込様式(神Excel)+ OnlyOffice の JavaScript + メール受付 |
| (フォーム JS の作成・保守) | North Mini Code(ローカルAI・開発時に生成) |
| メール送受信 | 自営メール(Stalwart) |

```
[公開層]  静的HTML → Cloudflare Pages(匿名・個人情報なし。案内と入口)
   ▲ build → 配信
[編集・運用層(機関内・自営)]  内容と文書を作る/共同編集する
   Forgejo(git。認証内蔵) / Zed(編集) / OnlyOffice Docs(エンジン。ホストはアプリ層)
   ▲
[アプリ層(機関内・自営)]  動的機能。認証・DB・メールを共有
   認証(PocketBase) / PostgreSQL / FastAPI / メール(Stalwart) / North Mini Code
   モジュール: 業務システム / 会員 / 決済連携(§13)/ 文書ホスト・受付処理
```

- 公開層は静的のまま。動的処理・認証・個人情報はすべて機関内に閉じる
- 公開ページからログインが要る機能へは、アプリ層の認証付き画面へ誘導する
- 必要度: Forgejo・Zed は共同編集の最低限。OnlyOffice Docs・
  North Mini Code はフォーム処理に有用。メールは自前(Stalwart)が望ましい
- 認証は層で分ける: 編集層は Forgejo 内蔵の認証・権限、アプリ層は
  PocketBase(基本的な認証のみ)。単一の ID 基盤(IdP)への統合は行わない
  (PocketBase は IdP 機能を持たず、統合には別途 IdP が要る。編集担当は
  少人数でありアカウントが二系統でも実害は小さい)
- 規模が小さければ subset で足りる(.md 直編集+ホスト型 git+既存メール)。
  規模に合わせて選ぶ

### 文書編集(OnlyOffice Docs+自作ホスト)

OnlyOffice Docs(Document Server)は保管・ユーザー管理を持たない編集エンジンで、
単体では動かず統合先(ホスト)が要る。既製のホスト(DocSpace・Nextcloud 等)は
使わず、**最小のホストを自作する**
(設計の詳細: https://aiseed.dev/ai-native-ways/software/documents/)。

- エンジンは Docs 単体を使う。DocSpace は LDAP/AD/SAML 前提で、切り離したはずの
  重い認証基盤が戻るため使わない(Docs 9.4 でエンジン単体の同時接続制限は撤廃。
  単一プロセスで動く)
- Nextcloud 等の保管アプリも足さない。ユーザー体系と DB が増え、構成の軽さと
  逆行する。必要なのは権限管理だけで、プラットフォーム全体は要らない
- ホストの三層: **実体**=ファイルシステム上のファイルそのまま(docx/xlsx の
  まま。AI・Polars 等が直接読める)/ **認証**=PocketBase(アプリ層と共有)/
  **権限**=構造化ストア(PostgreSQL。小規模は PocketBase 内の SQLite)
- 編集はファイル位置と保存先を JWT で署名して Docs に渡すだけ(数行)。
  同時編集の調停は Docs が担い、コールバックで保存が返る
- Office ファイルは「使う」のではなく「通過させる」: docx/xlsx は外部との
  交換形式(入口・出口)。社外共有は出口で PDF/docx にして送り、原本は
  機関内に置いたまま出さない

文書ホストは**アプリ層のモジュールの一つ**(認証は PocketBase)。ブラウザでの
文書編集にはアプリ層が要り、最小の共同編集(Forgejo+Zed)には含まれない——
§1 の運用形態の分岐と整合する。

## 3. コンテンツ設計

ページの性格で source を2種類に分ける。

| 種類 | 例 | source |
|---|---|---|
| 記事(散文) | お知らせ・報告・各種案内 | マークダウン + frontmatter(title/date/category) |
| 構成・反復データ | トップ・一覧・料金表・検索・事業者一覧 | YAML/JSON + テンプレート |

- 記事は担当者が .md を編集して更新する。frontmatter の category で
  一覧・お知らせへの掲載を制御
- トップの組版・表・一覧はデータ+テンプレートで生成する
- 判断に迷うページは記事(MD)側に倒す

ディレクトリ:

```
sites/<name>/
  site.yaml     サイト設定(base_url・分類ルール・担当→振り分け表)
  rules.py      サイト固有の抽出ルール(記事/一覧/表)
  source/       元データ(機関提供 or 取り込んだ生HTML: raw/)
  content/      記事 .md(編集の正)
  data/         構成・反復データ .yaml
  dist/         生成物(配信対象)
```

## 4. 生成・配信

- 生成: Jinja2 テンプレートで content/ と data/ から dist/ を作る
- 配信: cf-publish で dist/ を Cloudflare Pages / R2 へ反映する
  (cf-publish はキットに同梱。実際の公開実行は運用判断で行う)
- Cloudflare は檻ではない: 生成物はただの静的ファイルで、どのホスティングにも
  そのまま移せる。依存は配信だけで、内容(git)と道具(キット)は手元にある
- 並行運用: 移行期は別サブドメイン等で新旧を並走させる
- 切替: 確認後に切り替え、既存URLはリダイレクトで維持する

## 5. 問い合わせ処理

- 用途別(問い合わせ・依頼試験・設備利用 等)の申込様式(xlsx)で受ける。
  記入後、様式が組み立てる **`項目: 値` 形式の送信用テキストをメール本文へ
  貼り付けて**送信する。ファイルは添付しない(記入済み様式の添付は
  OnlyOffice 未使用時のフォールバック)。形式は厳密な YAML にしない——
  メールソフトの改行・整形に耐える、行単位の寛容なパーサで読む
  (seminar-kit と同じ方式)
- フォーム・受付アドレスは公開サイトに置かない(ボット収集・逆流対策)
- 担当は様式内のドロップダウンで選択する。受付アドレスは1つとし、受信側で
  「宛先(担当)」を読んで担当のフォルダ/転送へ振り分ける
- 送信者の実アドレスで届くため、メール2回入力・CAPTCHA は用いない
- 読み取れないものは自動で未処理キュー(受信フォルダ)へ落とし、人が処理する。
  本文はただのテキストなので安全に読める。フォールバックの添付は外部由来の
  ファイルとして扱い、人が確認するときはマクロ無効の環境(OnlyOffice の閲覧等)で
  開く(「開く=実行」を受信側で再導入しない)

### 様式(神Excel)が便利である理由——一つの定義が全チャネルを兼ねる

「神Excel」には悪評があるが、悪いのは VBA の同居・レイアウト依存の読み取り・
手作業の転記であって、様式そのものではない。それらを除いた xlsx 様式は、
フォーム定義として最も便利な部品になる:

- **様式=唯一のフォーム定義**: 一つの xlsx が、メール様式・ブラウザ記入
  (OnlyOffice Docs 経由)・印刷して紙/FAX の申込書と、全チャネルの定義を
  兼ねる。Web フォーム・PDF・紙様式を別々に作って別々に保守しない。
  受信側の読み取りも同一のパーサ(名前付きセル)に一本化される
- **相手に何も要求しない**: 記入者は使い慣れた表計算ソフト(Excel・
  LibreOffice・無料の OnlyOffice Desktop)で開いて書くだけ。アカウント
  発行・専用サイト・アプリのインストールが要らない
- **フォーム部品が最初から揃っている**: ドロップダウン(入力規則)・入力セル
  以外のシート保護・数式・レイアウト。Web フォームなら自作するバリデーションと
  画面を、表計算ソフトの標準機能で済ませられる
- **機械可読にできる**: 名前付きセルで読み取り位置を固定すれば、レイアウトを
  直しても読み取りは壊れない。見た目(人向け)と構造(機械向け)を
  1ファイルで両立する
- **検証がずれない**: OnlyOffice のマクロ(内部 JS)が送信前の未記入チェックと
  送信用テキストの組み立てを担う。チェックの定義はサーバー側の検証と同じ
  ソースから自動生成できるため、様式が複雑になってもクライアントとサーバーの
  検証が食い違わない(正の検証は常にサーバー側)
- **印刷に耐える**: 同じ様式がそのまま紙・FAX の申込書になる。デジタルと紙で
  二重の様式を持たない

この便利さは、xlsx が事実上の共通言語(誰の手元でも開ける・書ける・印刷
できる)であることに由来する。VBA を分離した(次節)うえでの神Excel は、
仮の回避策ではなく本格の部品である(§0)。実装例は seminar-kit(§13)。

### VBA を使わない設計(セキュリティの要)

様式の自動化は VBA ではなく、役割を2つに分離して実現する。これにより
「文書を開くこと自体がコードの実行になる」という VBA の攻撃経路を無くす。

- **外部 Python(処理)**: 本文テキスト・様式の読み取り、検証、登録を行う独立プログラム。
  文書の外にあり、git 管理・監査可能。ファイルを開いても起動しない
- **内部 JavaScript(補助)**: OnlyOffice のマクロ(サンドボックス内)。送信用
  テキストの組み立て等の入力補助のみで、システム権限を持たない。無くても
  数式・添付のフォールバックで成立する(OnlyOffice 未使用でも申込できる)
- データ(様式)と実行コードが同居しない。受信側の処理は常に外部 Python が
  行い、様式は保護とデータだけを持つ

手で書くなら、VBA も Python・JS も簡単ではない——書ける一部の人に属人化し
(「開かずのマクロ」)、しかも危険だった。この手書きの敷居は AI が下げ、
属人化はむしろ解消する(§7)。

## 6. メール

- 受付・自動返信・振り分けを担う。段階導入とする
  - 段階1: 既存のメール環境(SMTP/IMAP)を利用
  - 段階2: 自営メールサーバー Stalwart(SMTP/IMAP/JMAP・スパム対策・DKIM)へ移行。
    リスクの低い受信から自営化し、送信は到達性の検証が済んでから切り替える
- 送信は**自社回線の固定IP**から直接行う。自社の固定IPは、クラウドの共有IP帯
  (スパム送信に使われ評判が汚れがち)と違って送信履歴が汚れておらず、
  レピュテーションを自ら管理できる
- 送信切替前の確認項目: SPF/DKIM/DMARC・逆引き(PTR)の DNS 整備、
  主要プロバイダ宛の到達確認
- 送信処理・受信処理はそれぞれ1モジュールに集約する

## 7. ローカルAI(利用範囲を限定する)

AI は用途(コード/文書)と内容の機微(公開/内部秘)で使い分ける
(docs/05 と同じ二層)。基本はクラウド(Claude)、局所・内部秘だけローカル。

- コード: 基本は **Claude**(キット・生成器・パーサ・様式マクロの基盤・
  移行スクリプト=非機微)。細かい調整・現場の小さな自動化・内部データに
  触る分は **North Mini Code**(ローカル・VBA の代替)
- 文書: 基本は **Claude**(公開情報の変換・下書き)。内部秘・個人情報を
  含む分は **Command A+**(ローカル)
- 原則: 公開・非機微 → クラウド(Claude)/ 内部秘・個人情報・現場の局所
  → ローカルで建物の外に出さない

設置場所は限定する:

- 置かない: 公開物(静的HTML)、および申込→登録の決定的な処理経路
  (決定性・再現性の確保、受信メール経由のプロンプトインジェクション対策)
- 使ってよい(開発時): コードの基盤(パーサ・生成器・様式マクロ・移行
  スクリプト)は Claude、細かい調整・局所の自動化は North Mini Code。記事の
  HTML→Markdown 変換・文章の下書きは公開=Claude / 内部秘=Command A+。
  **出力(Python・JS・.md)は人が確認して git に凍結**し、公開物・処理経路には
  AI を残さない
- 使ってよい(運用): 記事・案内文の下書き、アンケート要約・返信文の下書き
  (Command A+。結果は人が確認してから用いる)

### North Mini Code の位置づけ(VBA の代替=誰でも安全な自動化を作れる)

VBA を外部Python+内部JavaScript に分ける設計(§5)は安全だが、手書きの
ハードルは高い。VBA も同じく簡単ではなく、書ける一部の人に属人化していた
(「開かずのマクロ」)。基盤のコード(パーサ・様式マクロ)は Claude が書き、
運用で必要になる現場の細かい調整・小さな自動化・内部データに触るコードは
North Mini Code が支える。要点は「AI が肩代わりする」ではなく、**専門家で
なくても、現場の担当が自分で安全な自動化を作れるようになる**こと。開発者採用
やベンダー委託(恒久コスト・依存)にも、特定の書ける人にも縛られない。

- 安全: ローカル実行。コードや実データを建物の外に出さない
- 低コスト: オープンウェイト(Apache)。席数課金・月額なし(電気代のみ)
- 裾野が広がる: かつて VBA で非プログラマが自作していた自動化を、いまは
  誰でも安全な Python・JS で作れる(「書ける人の異動で死ぬ開かずのマクロ」も、
  AI が支えることで属人化しない)
- あくまで**開発時の生成**。出力(Python・JS)は人が確認して git に凍結し、
  ランタイム・処理経路には AI を残さない(§5 の外部Python は AI ではなく
  凍結された通常のコード)
- かつて現場が VBA で作っていた自動化を、ローカルAI が生成する安全な
  Python/JS に置き換える、という位置づけ

### Command A+ の位置づけ(Markdown化・文章作成の鍵)

内容を Markdown で git 管理する本格の形(§3)も、同じ論理で AI が鍵になる。
CMS の WYSIWYG クリック編集に慣れた担当者には、Markdown を書くことも、旧CMS
の HTML を大量に Markdown へ移すことも、そのままでは難しい。ここを外注すれば
恒久コストと依存が戻る。Command A+ がその変換役・下書き役を担う。

- 安全: ローカル実行。内部秘・個人情報を含む文書も建物の外に出さない
  (公開情報はクラウド〔Claude 等〕で足りる=機微で使い分け。上記のとおり)
- 低コスト: オープンウェイト。席数課金・月額なし
- あくまで**開発・運用支援**。出力(.md)は人が確認して git に凍結し、公開物
  (静的HTML)や処理経路に AI を残さない(生成物はただの Markdown)
- 位置づけ: CMS の編集UI(の下書き・変換の手間)を、ローカルAI による
  Markdown 生成・整形に置き換える

**まとめ**: 本格の設計(VBA-free のフォーム、Markdown の内容)は手作業では
authoring の敷居が上がる。その敷居を AI が下げ、**専門家でなくても誰でも
作れるようになる**。機関は本格の形を自力で・安く・安全に持てる。

### 表計算を薄くする(データ=宣言的 / 計算=Python / 体裁=テンプレート)

Excel の多重関数リンクは、データ・ロジック・体裁をセルの中で絡ませる——壊れ
やすく・不透明・版管理もテストもできない。重い集計はピボットやアドインに
閉じ込められ、席数ライセンスと属人化を生む。**この分離を邪魔しているのが
Excel(表計算を"アプリ本体"にする使い方)**。移行後は表計算を薄くする:

- データ → **YAML/JSON**(宣言的・git 差分・行移動で壊れない。§3 の構成データ)
- 計算 → **Python**(監査・テスト可能・限界費用ゼロ。ピボット/アドインの代替)
- 体裁 → **テンプレート**(生成物として分離)
- 表計算(OnlyOffice)は「**記入・表示**」に軽くする。ロジックの置き場にしない

橋渡しはマクロ: OnlyOffice の JS マクロが記入済みシートを構造化 JSON/YAML に
変換する。**マクロは AI(North Mini Code)が開発時に生成し、runtime は決定的な
素の JS**(ランタイムに AI を置かない)。流れ: 記入(OnlyOffice)→ マクロ
(AI製・決定的)→ JSON/YAML → Python・テンプレート。CMS/Excel が一手に抱えて
脆くしていたものを、それぞれ得意な場所に返す。

## 8. サーバー構成・稼働

| 区分 | 稼働先 | 内容 |
|---|---|---|
| 公開サイト | Cloudflare Pages | 静的HTML。自サーバーに公開Webを置かない |
| 認証(アプリ層) | 機関内 | PocketBase。基本的な認証。職員・会員を役割で分離(モジュール利用時) |
| git 共同編集 | 機関内 | Forgejo(認証内蔵)。記事(.md)・データ(.yaml)の編集・版管理 |
| 編集クライアント | 担当者端末 | Zed(build/確認/配信をタスク化)・ブラウザ(OnlyOffice/Forgejo) |
| 文書・様式編集 | 機関内 | OnlyOffice Docs(編集エンジン)+ 自作文書ホスト(§2。認証は PocketBase) |
| メール | 機関内(段階2で Stalwart) | 送受信・受付・自動返信 |
| データ | 機関内(アプリ層) | PostgreSQL(モジュール利用時) |
| ロジック/API | 機関内(アプリ層) | FastAPI。受信処理・モジュール(モジュール利用時) |
| ローカルAI | 機関内 | North Mini Code 等。開発・運用支援(§7) |

- 公開層(Cloudflare)は常時稼働。編集・運用層/アプリ層は機関内の自営
- 機関内のサーバーはインターネットに直接面させない。公開が要る接点のみ限定
- 最小構成では編集・運用層とアプリ層を簡素にできる(規模で選ぶ)
- 移行完了後、旧CMS・公開Webサーバーは退役する
- 公開面のパッチ運用(CMS・PHP・Webサーバー)が不要になる。ただし機関内
  スタック(Forgejo・OnlyOffice・Stalwart・PocketBase・PostgreSQL 等)の更新は
  残る——攻撃面が公開Webから消える代わりに、運用は機関内に引き受ける(§9)

## 9. 運用手順

- 日常更新: content/ の .md を編集 → build → 確認 → 配信
- 共同編集: Forgejo(機関内 git)で複数担当が編集・レビュー・版管理。
  編集は Zed(build/確認/配信をタスク化)。文書・様式は OnlyOffice Docs でも編集可
- 版管理: git(誰がいつ何を変えたかを履歴で追える)
- サーバー更新: 機関内スタックは各サービスをコンテナで持ち、定期(月次目安)に
  一括更新する。更新は一つのスクリプト(またはタスク)にまとめ、手順書に含める。
  機関内サーバーはインターネットに直接面していない(§8)ため、公開Webの
  パッチ運用のような緊急性はない
- 監視: **公開Webとメールだけ**簡易な監視を置く。この二つは止まっても
  「外から来るものが来なくなるだけ」で内側からは気づきにくいため——公開URLの
  外形確認と、自分宛て送受信の疎通確認を定期実行して失敗を通知する、小さな
  スクリプトで足りる。編集層・アプリ層(Forgejo 等)は止まれば使う人がすぐ
  気づくので、専用監視は置かない
- バックアップ: ソース・記事は git(Forgejo のリポジトリを外部へ複製)。
  PostgreSQL はダンプ、メールストア・文書ホストの実体ファイルと権限ストアは
  ファイル複製で、いずれも機関内の別筐体へ定期取得する。受信原本・受付データも
  同様に機関内で保全
- 引き継ぎ: 「編集→生成→配信」の手順書を成果物に含める。担当者が
  一巡できることを到達目標とする

## 10. 移行手順

**順序の要:**

- **まず共同編集の基盤(Forgejo+Zed)を立てる**——CMS の核=複数人編集の
  置き換え。内容はここに載せて複数担当で編集する(§1)
- **内容の入口は「公開分を Claude で Markdown 化」**。公開情報は外に出る内容
  なのでクラウド(Claude)で変換してよい。効果確認・デモは1人でも小さく始められる
- 内部秘の変換(Command A+)・問い合わせ様式・アプリ層モジュールは、
  必要になった段階で足す(§7 の機微による使い分け・§13)

以下は全体の順序:

1. 共同編集の基盤(Forgejo+Zed)を立てる。内容データは機関側で用意する
   (CMSエクスポート・既存原稿等)。
   データがあればクロールは行わない。クロールは了解を得たうえでの最後の手段
2. 記事=MD / 構成=データ に変換する(機械変換+人の仕上げ)。HTML→Markdown の
   下書き・抽出/変換スクリプトには AI を使う(機微による使い分けと git 凍結の
   原則は §7)
3. テンプレートで生成し、並行公開して確認する
4. 切替後、既存URLはリダイレクトで維持する
5. 旧CMSを退役する(保守・ライセンス・脆弱性対応の負担が解消する)

## 11. セキュリティ

- 攻撃面: 公開面から動的CMS・PHP・フォーム処理を除去する
- 配信: Cloudflare が TLS・WAF・DDoS 対策・セキュリティヘッダ(HSTS/CSP 等)を
  担う。HTTP/2/3 も自動
- 個人情報: 公開サイト・外部クラウドに置かない。問い合わせ内容はメールで
  機関内に届き、機関内で保全する
- 真正性: 送信者アドレスで本人性を確認する
- 様式の自動化に VBA を使わない: 外部Python(処理)+内部JavaScript(サンドボックス)
  に分離し、データと特権実行コードを同居させない(§5)。「開く=実行」の攻撃面がない

## 13. アプリ層とモジュール(業務システム・決済・会員機能)

アプリ層は、認証・DB・メールを共有基盤とし、その上にモジュールを載せる。
公開層(静的)は変えない。機関内で稼働し、インターネットに直接面させない。

### 共有基盤(モジュールの土台)

- **認証(PocketBase)= アプリ層の必須コンポーネント**。基本的な認証のみを
  担い、職員(業務システム操作)と事業者会員(会員ポータル)を**役割(role)で
  権限分離**する。アプリ層のモジュールは独自にログインを作らず、これを使う。
  編集層(Forgejo)は内蔵の認証を使い、この基盤には統合しない(§2)
- **データ(PostgreSQL)**: モジュールはスキーマを分けて同一DBに同居できる
- **メール**: 通知・受付・自動返信を共有(§6)
- 個人情報・認証情報はこの層(機関内)に閉じ、公開層・外部クラウドに出さない

### モジュール

| モジュール | 例 | 設計 |
|---|---|---|
| 業務システム | 研修・セミナー管理(申込→名簿→台帳)、依頼試験・設備予約の受付管理 | 事務局アプリ(FastAPI/DB直結、UI は Flet 等)。認証は共有基盤。**seminar-kit がこの形の実装例**(職員認証に PocketBase を使用) |
| 会員(事業者ポータル) | 事業者のログイン・申請状況照会・継続利用 | 認証は共有基盤(会員 role)。公開層からはログイン画面へ誘導するだけ。個人データはアプリ層に閉じる |
| 文書ホスト | 社内文書の保管・ブラウザ編集(OnlyOffice Docs 統合) | 自作の最小ホスト(§2)。実体=ファイルシステム、権限=構造化ストア、認証は共有基盤 |
| 決済連携 | 受講料・手数料の収納 | カード情報は自組織で保持せず**外部の決済事業者に委譲**(PCI DSS 範囲を持ち込まない)。可能なら請求書・公金収納(証紙代替の指定納付等)に寄せる。アプリ層は収納状況のみ保持 |

### 原則

- **公開層は静的・匿名・個人情報なし**を保つ。動的性・認証・個人情報は
  すべてアプリ層(機関内)に閉じ込める
- **認証はアプリ層の基盤**であり、モジュールごとに作らず共有する
- モジュールの追加はアプリ層への追加であって、公開層(§2〜§5・§8〜§12 の
  静的部分)の設計は変わらない
- 共有基盤は「既知の要件のための土台」として持つ。個々のモジュールの内部に
  憶測の拡張点・汎用設定は作らない(必要になったモジュールを単純に書く)
