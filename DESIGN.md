# 設計書 — 組織のIT移行(Microsoft 365 + Azure・業務システム・CMS → 開いた自営スタック)

組織のIT——Microsoft 365 + Azure(文書・メール・会議・認証・DB・実行基盤)・
業務システム・CMS(公開Web)
——を、開いた自営スタック(静的配信・共同編集・問い合わせ・業務)へ移行する
ための設計書。サーバー運用の担当者が構築・運用・引き継ぎに用いる。移行と
生成のツール一式は aiseed-migration-kit(本リポジトリ)を使う。

## 0. 設計の考え方(CMS の「仮」を本格にする)

CMS は、本体で扱えない業務を仮の手段へ押し出す。依頼試験・設備予約・研修は
担当PCの**その場しのぎの Excel**・紙・メールで処理され、フォームは後付け
プラグイン、検索や解析は外部サービス頼み、内容編集は「ログインして直す」
だけで承認も版管理も薄い。結果として業務が分断・属人化し、CMS 自身の機能も
後付けの仮になりがち。

本構成は、その仮を一つずつ本格の形に引き上げる:

- 内容 → git で版管理・共同編集・承認(本格の編集プロセス)
- 申込・問い合わせ → 様式プロファイル(AsciiDoc。§5)を唯一の定義とする
  機械可読なデータ経路(神Excel はそこから派生する記入チャネル)。
  **これは仮ではなく本格の部品**——VBA(文書内に特権実行コードが同居)を、
  外部Python(処理)+内部JavaScript(サンドボックスの補助)に分離し、
  「開く=実行」という攻撃経路を設計から無くしている(§5)
- 業務(試験・予約・研修) → 認証・DB を備えたアプリ(その場しのぎ Excel・紙の置換)
- 認証 → アプリ層で職員・会員を role 分離(PocketBase)。編集層は Forgejo の内蔵認証
- メール → 自営で受付・振り分け(代表フォーム一本の置換)

狙いはコスト削減以上に、**仮の回避策を無くし、所有・統合・監査できる形に
すること**。ただし本格化は運用の引き受けを伴う(§1 の運用形態)。全業務を
無条件に本格化するのではなく、業務ごとに要否で水準を選ぶ(単独・小規模の
部分は静的化までで十分な場合がある)。

### 三つの閉じた世界からの開放(Microsoft 365 + Azure・業務システム・CMS)

機関の情報環境は、三つの閉じた世界に分かれてきた——**Microsoft 365 + Azure**
(文書・メール・会議に加え、認証・データベース・実行基盤。席数課金・
従量課金・外部クラウド)、**業務システム**(基幹・業務パッケージ。
独自形式・独自API・ベンダー保守契約)、**CMS**(公開Web。
ライセンス・保守・脆弱性対応)。それぞれが独自形式と契約で閉じ、データは
外に出しにくく、世界の境界のたびに転記と受け渡しが生じる(冒頭の
「分断・属人化」の正体)。

本構成は、この三つを**一つの開いた土台**(認証・SQL・OnlyOffice・メール・
git・静的公開)で置き換える。公開Web側から見れば「CMS移行キット」、業務・
文書・メール側から見れば「Microsoft 365 移行キット」——同じ道具の顔が違う
だけである。業務システムは同じ土台の上のモジュールになる(§13)。
Microsoft 365 + Azure との対応:

| Microsoft 365 + Azure | 本構成(自営) |
|---|---|
| Microsoft Entra ID(認証・SSO) | 各システムの内蔵認証(Forgejo・PocketBase・Stalwart。§2「認証を取り戻す」) |
| Exchange(メール・予定表) | Stalwart(SMTP/IMAP/JMAP・CalDAV)+ Thunderbird |
| Word/Excel・Office 文書 | OnlyOffice |
| OneDrive / SharePoint(文書共有) | 自作文書ホスト + OnlyOffice Docs(§2) |
| Teams(会議・ウェビナー) | Jitsi / BigBlueButton |
| Teams(チャット) | **対象外**——効率が悪く、決定・資料が流れて消える。連絡=メール / 議論=Forgejo(issue)/ 会議=Jitsi で足りる |
| Bookings(予約) | Cal.com |
| Azure SQL / Cosmos DB(データベース) | PostgreSQL / SQLite(製品ではなく SQL にコミット。§0「標準でつなぐ」) |
| Azure App Service / Functions(業務ロジックの実行) | FastAPI(機関内サーバー) |
| Azure DevOps(リポジトリ・CI/CD) | Forgejo |
| Azure Blob Storage(ファイル保管) | 機関内のファイルシステム(文書ホスト §2)。公開配信のみ Cloudflare(Pages/R2) |
| Power BI(集計・分析) | DuckDB / Polars + Python(帳票・グラフは生成物として出力) |
| Copilot(AI) | ローカルAI(Command A+ / North Mini Code) |
| VBA / Power Automate(自動化) | 外部Python + 内部JavaScript |
| SharePoint / CMS(公開) | 静的HTML(Cloudflare)+ Forgejo(git) |
| 席数課金・従量課金のクラウド | 自営 OSS(継続利用料なし) |

部品の選定は自立編スタック(https://aiseed.dev/ai-native-ways/software/)に
よる。会議・ウェビナー・予約は必要になった機関で足す(本キットの必須部品では
ない)。

なお M365 の閉じ方は一様ではない。**文書そのものは今日でもローカルの
Office で境界内に置ける**。閉じ込めの本丸は認証(Entra ID)・メール
(Exchange Online)・共有(SharePoint/OneDrive)——ローカルの代替が提供
されない部分である。そして Copilot が、残った文書を雲へ吸い上げる次の
一手になる: Copilot は相手の境界内のデータしか読まないため、「AI に仕事を
させたければ文書をクラウドへ」という圧力が利便の形で働く。本設計の
文書ホスト(§2)が「実体はファイルのまま・AI が直接読める」を原則に
するのは、Copilot 相当をローカルで実現し、この圧力を無効にするためである。

いずれも「ベンダーの檻から手元へ」——所有・限界費用ゼロ・データを建物の中に、
という同じ原則の適用先が違うだけである。

### 閉じた世界をつくらない(標準でつなぐ)

目的は、三つの閉じた世界を**別の閉じた世界(自前スタックへの新たな
ロックイン)に置き換えることではない**。部品は独自形式・独自 API でなく、
少数の公開標準で疎結合に繋ぐ:

- データ = **Markdown / YAML / JSON / Parquet**(自己記述・スキーマ付き。
  位置がずれると黙って壊れる CSV は境界に使わない)
- 問い合わせ・蓄積 = **SQL**(実装は PostgreSQL・SQLite。製品ではなく
  SQL にコミットする)
- 処理・接着 = **Python**、体裁 = テンプレート(Jinja2)

自己記述で・AI が扱えて・多くの道具が読める形式だから、部品は差し替え
可能で、データはツールより長生きし、**この仕組み自体からも出られる**。
檻から出ることと、出た先を檻にしないこと——その両方で「開放」である。

### Claude の役割(開放を現実にする)

OSS はもともと無料だった。三つの世界に機関を閉じ込めてきたのは、OSS を
組み立て・維持する労力と専門性である。**Claude がその労力を担うことで、
自営=開放が初めて現実的になった**。

- 担うもの: 構築(設定・接着コード)、移行(内容の変換・下書き)、保守
  (調整・改修のコード化)。人に残るのは確認・判断・内容の最終責任で、
  手作業に比べ時間は桁違いに小さい(ただしゼロではない)
- 席は**運用責任者の1席**で足りる。ほかの職員は無償の OSS(Forgejo・Zed・
  OnlyOffice)と共有のローカルAI(North Mini Code・Command A+)で完結する。
  M365 の「席数 × 全員」とは逆で、**費用が利用者数に比例しない**
- 内部秘・個人情報・現場の局所はローカルAI と分担する(使い分けは §7)
- **Claude 自体も檻にしない**: 使うのは開発・移行・保守のときだけで、出力
  (Python・JS・Markdown・YAML)は人が確認して git に凍結し、ランタイム・
  処理経路には残さない。成果物はすべて公開標準の形式なので、Claude を
  やめても仕組みはそのまま動き、別の道具でも保守を続けられる

正直な一線: これは「今こそ現実的になった」であって楽勝ではない。正しさの
検証・段階移行・最終責任(機関)は残る。AI は労力を桁違いに下げるが、
判断をゼロにはしない。

### 最大の利点: CMS と Microsoft 365 を一つの移行で

普通は別プロジェクトになる「CMS 脱却」「Microsoft 365 脱却」「業務システムの
置き換え」が、**土台を共有するため一つの移行に畳める**。サーバースタック
(PostgreSQL・メール・OnlyOffice・git・認証)・ローカルAI が、Web 側と業務側の
両方を担う。OnlyOffice は申込様式の編集と日常の Office 文書を、メールは
問い合わせ受付と職員メールを、同じ基盤で兼ねる。結果、二重投資・二重運用・
両者の継ぎ目が生じない。

「一気に」の意味は**基盤を一度に作る**こと。切替は段階でよい(公開分→
問い合わせ様式→文書・メール→業務モジュール)。基盤が共有なので、段階を
踏んでも二重にならない。

### 一つになることで業務が流れる

別々のシステム(CMS・M365・担当PCの Excel)は、境界ごとに手作業の転記と
受け渡しを生む。告知は CMS、申込はフォーム、名簿は Excel、報告は Word、
連絡は Exchange——データが繋がらず、境界のたびに入れ直す(§0 冒頭の
「分断・属人化」の正体)。一つの土台(共通の認証・データ・道具)に載ると、
この境界が消えて業務が一本の流れになる。

- 告知 → 申込 → 処理 → 名簿 → 実績 → 連絡 が一つのデータの流れになる
  (再入力なし。申込様式がそのまま名簿・台帳・受領メールへ流れる)
- 認証が一つなので記録が繋がる(同じ事業者の試験・設備・研修の履歴が一望)
- 文書・データ・メールが同じ基盤にあり、Web→様式→DB→文書→メールが
  継ぎ目なく動く

コスト・所有の先にある本質的な価値は、この**業務が流れる(分断・属人化・
再入力が消える)**ことである。

### そのうえ、ソフトは無料

本格化・業務の流れ・一つの移行——それを支える構成部品が、すべて
OSS / オープンウェイトである。**ソフトのライセンス料・席数課金・サブスクは
ゼロ**(a-blog 等の CMS ライセンスも M365 の席数課金も消える)。

| 部品 | ライセンス |
|---|---|
| Cloudflare Pages | 無料枠(公開静的で足りる) |
| Forgejo / OnlyOffice | AGPL |
| Stalwart / PocketBase / PostgreSQL | OSS |
| North Mini Code(Apache)/ Command A+ | オープンウェイト |
| Python / Jinja2 / git / Zed | OSS/無償 |

無料なのは**ソフト**である。残る費用は、ハードウェア(サーバー類は多くが既存
流用。ただしローカルAI は別で、North Mini Code は GPU 1枚で動くが、Command A+ は
大型モデルのため相応の GPU メモリへの一回性の投資が要る)・
電気代・構築と移行の一回性の手間・運用の人手・任意の運用支援。いずれも
利用者数に比例して恒久的に増える種類のものではない(限界費用ゼロに近い)。

そのうえ、**構築と移行の手間の大半は AI が担い、要るのは運用責任者の
1席だけ**である(上記「Claude の役割」)。結果、恒久的に増える費用はほぼ
残らない。

**構築も速い。** 部品がほぼ OSS なので、調達・ライセンス交渉・ベンダーの
見積待ち・契約というリードタイムがない(プロプライエタリなら数ヶ月かかる
ところ)。部品は標準的でコンテナ化されており、設定・接着コードは AI が書く。
——ただし速いのは基盤の立ち上げで、時間がかかるのは内容の変換と検証
(メール到達性・認証・様式の動作確認)。そこは段階的に、AI で加速しながら
進める(§10)。

### データの置き場(対応表)

保管庫は性質ごとに固定し、これ以上増やさない。NoSQL・オブジェクトストレージ・
メッセージキュー・キャッシュ層は導入しない。

| データの性質 | 正の置き場 | 備考 |
|---|---|---|
| 判断(様式定義・決裁文書・規約・DDL 生成元) | git(テキスト) | 差分・署名・PR(§5・§14) |
| 記録(申込・入金・承認などの業務イベント) | PostgreSQL | 型・制約・トランザクション。git 上の定義から再生成できる派生物(§5) |
| 証跡(受信メール原文・添付) | Stalwart IMAP | 改変しない(§6) |
| 交付の凍結 | PDF/A | 生成元(コミットハッシュ)と交付物ハッシュを併記(§14) |
| 分析用アーカイブ(締め済みデータ・ログ) | Parquet | DuckDB で直接読む。DB を肥大させない |
| 観測(センサー・気象等の多次元データ) | netCDF(CF 規約) | 将来拡張。現行キットでは未使用 |

「正はテキスト+git」(冒頭)の具体化であり、PostgreSQL・Parquet 等はすべて
git 上の定義から決定的に再構築できる派生物という位置づけを崩さない。

## 1. 目的・範囲

- 公開サイトを静的HTML化し、Cloudflare Pages で配信する
- 内容(ソース)を git 上のマークダウンとデータで管理する
- 問い合わせフォーム(唯一の動的機能)をメール受付に置き換える
- 対象は公開情報の提供サイト。個人情報・認証を要する処理は機関内に閉じる
- **二層構成**とし、業務システム・決済・会員機能を載せられる土台を最初から持つ:
  - **公開層**(静的・匿名・個人情報なし。Cloudflare Pages)
  - **アプリ層**(機関内。認証・DB・メールを共有し、その上にモジュールを載せる)
- **認証(ID基盤)はアプリ層の必須コンポーネント**。業務・会員・決済の
  いずれも認証を前提とする(§13)
- 最小構成(案内サイト+メール問い合わせのみ)ではアプリ層は不要。
  モジュールを持つ時にアプリ層を立てる。使わない柔軟性を先に作り込むのではなく、
  モジュールが載る場所と認証土台を設計として定めておく、という趣旨

### 運用形態で構成が決まる(最重要の分岐)

構成の大きさは規模ではなく**運用形態=共同編集の有無**で決まる。

- **1人運用(単独更新)**: 静的サイト化だけで足りる。`.md` を直接編集 →
  build → 配信。Forgejo・OnlyOffice Docs・アプリ層は不要
- **共同編集(複数人)**: 「誰が編集し・誰が承認し・履歴をどう突き合わせるか」が
  必要になる——これは CMS が担っていた機能そのもの。よって
  Forgejo(git 共同編集。認証・権限は Forgejo 内蔵)+ Zed(+ OnlyOffice Docs)が
  要る。業務・会員・決済を持てばアプリ層も加わる。**結局スタック全体が必要になる**

多くの機関では内容を複数人で編集する。CMS の核はまさにこの「複数人編集」で
あり、置き換えで**最初に据えるのは共同編集の基盤=Forgejo(git)+Zed**である。
内容はここに載せて編集し、静的生成・配信・問い合わせ
様式・アプリ層モジュールはその上に順次足す。1人運用に限れば基盤を省き
静的化だけでよい(小規模の例外)。段階的に上げられ、土台は変えず上に積む。

## 2. 全体構成

CMS が一体で担っていた機能を、役割ごとに自営の道具へ分解する。公開層だけが
外(Cloudflare)に出て、編集・運用とアプリは機関内で自営する。すべて自営
OSS で、継続ライセンス料はない。

| CMS が担っていた機能 | 置き換え |
|---|---|
| 公開ページの配信 | 静的HTML + Cloudflare Pages(公開層) |
| 複数人での内容編集・版管理 | Forgejo(git 共同編集。認証内蔵)+ Zed(編集) |
| 文書・様式の編集 | OnlyOffice Docs(編集エンジン)+ 自作文書ホスト(ブラウザ編集) |
| 問い合わせフォーム処理 | 様式プロファイル(§5)から派生する申込様式(xlsx)+ OnlyOffice の JavaScript + メール受付 |
| (フォーム JS の作成・保守) | North Mini Code(ローカルAI・開発時に生成) |
| メール送受信 | 自営メール(Stalwart) |

```
[公開層]  静的HTML → Cloudflare Pages(匿名・個人情報なし。案内と入口)
   ▲ build → 配信
[編集・運用層(機関内・自営)]  内容と文書を作る/共同編集する
   Forgejo(git。認証内蔵) / Zed(編集) / OnlyOffice Docs(エンジン。ホストはアプリ層)
   ▲
[アプリ層(機関内・自営)]  動的機能。認証・DB・メールを共有
   認証(PocketBase) / PostgreSQL / FastAPI / メール(Stalwart) / North Mini Code
   モジュール: 業務システム / 会員 / 決済連携(§13)/ 文書ホスト・受付処理
```

- 公開層は静的のまま。動的処理・認証・個人情報はすべて機関内に閉じる
- 公開ページからログインが要る機能へは、アプリ層の認証付き画面へ誘導する
- 必要度: Forgejo・Zed は共同編集の最低限。OnlyOffice Docs・
  North Mini Code はフォーム処理に有用。メールは自前(Stalwart)が望ましい
- 認証は層で分け、単一の ID 基盤(IdP)への統合は行わない(下記
  「認証を取り戻す」)
- 規模が小さければ subset で足りる(.md 直編集+ホスト型 git+既存メール)。
  規模に合わせて選ぶ

### 認証を取り戻す(個別のシステムでチェックする)

実装で最も重要な部品は認証である。三つの閉じた世界は、いずれも認証を外部の
ID 基盤(Microsoft Entra ID 等)に預けることで成立している——データを
建物の中に置いても、**鍵が外にあれば閉じた世界のまま**である。席数課金・
外部側の障害・仕様変更が、全システムの入口を一度に左右する。

本構成は認証を手元に取り戻す。そして、取り戻した認証を自前の単一 SSO に
集約することもしない——**個別のシステムが自分でチェックする**:

- 編集層: Forgejo 内蔵の認証・権限
- アプリ層のモジュール(業務・会員・文書ホスト): PocketBase
  (基本的な認証のみ。職員・会員を role 分離。§13)
- OnlyOffice Docs: 文書ホストが発行する JWT(上記「文書編集」)
- メール: Stalwart 内蔵のアカウント

単一の IdP は便利だが、単一の依存点・障害点になる——外部 ID に預けていた
構図を自前で再現するだけである(PocketBase は IdP 機能を持たず、統合には
別途 IdP を運用に足すことになる、という実装事情とも一致する)。個別
チェックなら、どれか一つが止まっても他は動き、部品は認証ごと独立に
差し替えられる——「閉じた世界をつくらない」(§0)の認証版である。
少人数の運用では、アカウントが二系統になる手間より、依存が消える利益が
大きい(冒頭の運用形態の分岐と同じ判断)。

### 文書編集(OnlyOffice Docs+自作ホスト)

OnlyOffice Docs(Document Server)は保管・ユーザー管理を持たない編集エンジンで、
単体では動かず統合先(ホスト)が要る。既製のホスト(DocSpace・Nextcloud 等)は
使わず、**最小のホストを自作する**
(設計の詳細: https://aiseed.dev/ai-native-ways/software/documents/)。

- エンジンは Docs 単体を使う。DocSpace は LDAP/AD/SAML 前提で、切り離したはずの
  重い認証基盤が戻るため使わない(Docs 9.4 でエンジン単体の同時接続制限は撤廃。
  単一プロセスで動く)
- Nextcloud 等の保管アプリも足さない。ユーザー体系と DB が増え、構成の軽さと
  逆行する。必要なのは権限管理だけで、プラットフォーム全体は要らない
- ホストの三層: **実体**=ファイルシステム上のファイルそのまま(docx/xlsx の
  まま。AI・Polars 等が直接読める)/ **認証**=PocketBase(アプリ層と共有)/
  **権限**=構造化ストア(PostgreSQL。小規模は PocketBase 内の SQLite)
- 編集はファイル位置と保存先を JWT で署名して Docs に渡すだけ(数行)。
  同時編集の調停は Docs が担い、コールバックで保存が返る
- Office ファイルは「使う」のではなく「通過させる」: docx/xlsx は外部との
  交換形式(入口・出口)。社外共有は出口で PDF/docx にして送り、原本は
  機関内に置いたまま出さない

文書ホストは**アプリ層のモジュールの一つ**(認証は PocketBase)。ブラウザでの
文書編集にはアプリ層が要り、最小の共同編集(Forgejo+Zed)には含まれない——
§1 の運用形態の分岐と整合する。

## 3. コンテンツ設計

ページの性格で source を2種類に分ける。

| 種類 | 例 | source |
|---|---|---|
| 記事(散文) | お知らせ・報告・各種案内 | マークダウン(または AsciiDoc) + frontmatter(title/date/category) |
| 構成・反復データ | トップ・一覧・料金表・検索・事業者一覧 | YAML/JSON + テンプレート |

- 記事は担当者が .md(または .adoc)を編集して更新する。frontmatter の
  category で一覧・お知らせへの掲載を制御。.adoc は pyasciidoc(CJK対応の
  見出し・強調。§7 と同じ理由でクラウドの Claude が開発・保守を担う)が
  無いとビルド時に分かりやすいエラーになる(§0 の「仮ではなく本格の部品」
  の考え方どおり、黙ってフォールバックしない)
- トップの組版・表・一覧はデータ+テンプレートで生成する
- 判断に迷うページは記事(MD)側に倒す

ディレクトリ:

```
sites/<name>/
  site.yaml     サイト設定(base_url・分類・担当→振り分け表・様式への参照)
  forms/        様式プロファイル .adoc(様式の唯一の定義。§5)
  rules.py      サイト固有の抽出ルール(記事/一覧。任意)
  source/       元データ(機関提供 or 取り込んだ生HTML: raw/)
  content/      記事 .md(編集の正。convert は既存を上書きしない)
  data/         構成・反復データ .yaml
  templates/    テンプレートの上書き(任意)
  public/       そのまま配信するファイル(favicon・_redirects 等。任意)
  forms-out/    生成した様式(xlsx・記入用テキスト。build が dist/forms/ に載せる)
  dist/         生成物(配信対象)
```

## 4. 生成・配信

- 生成: Jinja2 テンプレートで content/ と data/ から dist/ を作る
- 配信: cf-publish で dist/ を Cloudflare Pages / R2 へ反映する
  (cf-publish はキットに同梱。実際の公開実行は運用判断で行う)
- Cloudflare は檻ではない: 生成物はただの静的ファイルで、どのホスティングにも
  そのまま移せる。依存は配信だけで、内容(git)と道具(キット)は手元にある
- 並行運用: 移行期は別サブドメイン等で新旧を並走させる
- 切替: 確認後に切り替え、既存URLはリダイレクトで維持する

## 5. 様式プロファイルと問い合わせ処理

### 様式プロファイル(正の転換)

様式の唯一の定義は **AsciiDoc 文書(様式プロファイル)** である。神Excel
(xlsx)は定義ではなく派生物に降格する。site.yaml の inquiry.forms は
AsciiDoc 様式プロファイルへの参照に置き換える。一つの様式定義から
以下を派生させる(生成順は §5「派生生成」):

1. 記入用 xlsx(openpyxl。マクロは従来通り OnlyOffice JS を別出力)
2. 記入用プレーンテキスト(画面コピペ・メール返信チャネル用。定義文書
   自体が配布物になる)
3. Pydantic 検証モデル
4. PostgreSQL DDL(CREATE TABLE。§0「データの置き場」の記録層)
5. AI 用プロンプト(pending 解釈用。§7。制約が機械的に列挙されるため
   プロンプトとサーバー検証が食い違わない。LLM はオプショナルで、
   無くても pending は人が処理できる)

### 構文の規律

独自の派生言語は作らない。AsciiDoc の属性リスト・ラベル付きリスト
(`項目:: `)の範囲内で語彙を定義する。標準の AsciiDoc プロセッサ
(Asciidoctor 等)に食わせても、未知属性が無視されて「人間が読める記入
様式」としてレンダリングされること(後方互換の保証)。レンダリング
(CJK 対応)とプロファイル解釈(様式)は分離する——レンダリングは
pyasciidoc(別パッケージ)、プロファイル解釈は受付基盤に単独で組み込める
キット内の小さな独立モジュール(amig.inquiry.profile。markdown-it-py の
トークン列に依存しない)。

### 型と制約 = SQL 語彙

フィールドの型・制約は SQL の語彙をそのまま使う。独自の型体系を発明しない。

- 例: `[varchar(50), not null]` / `[date, not null]` /
  `[integer, check (参加人数 between 1 and 20)]` / 選択肢はリスト列挙 →
  enum または CHECK IN / `[references 事業者(番号)]` → 外部キー /
  明細の繰り返し(受講者一覧等)→ 子テーブル(1:N)
- メール形式等は CHECK + 正規表現で制約群に収める
- 表示層(記入例・説明文・複数行か否か等)は**属性を規格化しない**。普通の
  日本語で本文に書く。xlsx 生成・テキスト整形時に AI が文脈ごと解釈して
  配置する。hint / placeholder のような表示用属性は導入しない

### 正規化と修復

正規化の独立した方針レイヤーは設けない。正規化は「制約を満たす形への
決定的な修復」として制約から導出する。修復関数は制約タイプへの対応表として
実装する(制約の随伴物)。

- 全角カナ制約 → NFKC + ひらがな→カタカナ
- integer → 全角数字→半角
- date → 和暦・区切り揺れの解釈

制約から一意に導出できない入力(カナ欄に漢字等)は修復せず pending へ。
修復が入った投稿は原文を証跡として保持する(受信メール原文は IMAP に残る。
保存値は制約準拠、原文は証跡。§0「データの置き場」)。

### 受信パーサーの寛容性(非対称)

配布は AsciiDoc として厳密に、受信は「AsciiDoc 風テキスト」として寛容に。
想定する揺れ: 全角コロン(`項目:値`)、NBSP 混入、format=flowed の自動
改行、HTML メール化によるマークアップ喪失、行の削除・順序入れ替え。
処理順: NFKC 正規化 → ラベル行の緩い正規表現マッチ → 制約検証
(Pydantic)→ 検証落ちは pending フォルダへ。

### 受信・送信の運用

- 用途別(問い合わせ・依頼試験・設備利用 等)の申込様式で受ける。記入
  チャネルは2つ、どちらも同じ様式プロファイルから派生する:
  (1) xlsx——記入後、様式が組み立てる **`項目: 値` 形式の送信用テキストを
  メール本文へ貼り付けて**送信する。ファイルは添付しない(記入済み様式の
  添付は OnlyOffice 未使用時のフォールバック)。
  (2) 記入用プレーンテキスト——画面からコピーして記入し、そのままメール
  本文として送る(マクロレスの最小依存経路)。
  形式は厳密な YAML にしない——メールソフトの改行・整形に耐える、
  行単位の寛容なパーサで読む(seminar-kit と同じ方式)
- **様式は公開サイトからダウンロードできる**(build が dist/forms/ に載せる。
  これが無いと初回の接点が電話に限られ、問い合わせの置き換えにならない)。
  ただし**受付アドレスはページ本文に書かず、様式の中にだけ**置く(HTML 上の
  mailto をボットに収集させない)。ボット・スパムへの実対策は秘匿ではなく
  挙動で担保する——様式由来でないメールには自動返信しない(バック
  スキャッター防止)、読めないものは未処理キューで人が判断する。
  受領メール・印刷物・電話での個別送付も引き続き使える
- 担当は様式内のドロップダウンで選択する。受付アドレスは1つとし、受信側で
  「宛先(担当)」を読んで担当のフォルダ/転送へ振り分ける
- 送信者の実アドレスで届くため、メール2回入力・CAPTCHA は用いない
- 読み取れないものは自動で未処理キュー(受信フォルダ)へ落とし、人が処理する。
  本文はただのテキストなので安全に読める。フォールバックの添付は外部由来の
  ファイルとして扱い、人が確認するときはマクロ無効の環境(OnlyOffice の閲覧等)で
  開く(「開く=実行」を受信側で再導入しない)

### 様式(神Excel)が便利である理由——一つの定義から全チャネルを派生させる

「神Excel」には悪評があるが、悪いのは VBA の同居・レイアウト依存の読み取り・
手作業の転記であって、様式そのものではない。それらを除いた xlsx 様式は、
様式プロファイル(AsciiDoc)から派生する記入チャネルとして最も便利な
部品になる:

- **様式=様式プロファイルからの派生物**: 唯一の定義は AsciiDoc(前節)で、
  xlsx はそこから生成される記入チャネルの一つ。メール様式・ブラウザ記入
  (OnlyOffice Docs 経由)・印刷して紙/FAX の申込書と、全チャネルが同じ
  定義から生成される。Web フォーム・PDF・紙様式を別々に作って別々に
  保守しない。受信側の読み取りも同一のパーサ(名前付きセル)に一本化される
- **相手に何も要求しない**: 記入者は使い慣れた表計算ソフト(Excel・
  LibreOffice・無料の OnlyOffice Desktop)で開いて書くだけ。アカウント
  発行・専用サイト・アプリのインストールが要らない
- **フォーム部品が最初から揃っている**: ドロップダウン(入力規則)・入力セル
  以外のシート保護・数式・レイアウト。Web フォームなら自作するバリデーションと
  画面を、表計算ソフトの標準機能で済ませられる
- **機械可読にできる**: 名前付きセルで読み取り位置を固定すれば、レイアウトを
  直しても読み取りは壊れない。見た目(人向け)と構造(機械向け)を
  1ファイルで両立する
- **検証がずれない**: OnlyOffice のマクロ(内部 JS)が送信前の未記入チェックと
  送信用テキストの組み立てを担う。チェックの定義はサーバー側の検証と同じ
  ソースから自動生成できるため、様式が複雑になってもクライアントとサーバーの
  検証が食い違わない(正の検証は常にサーバー側)
- **印刷に耐える**: 同じ様式がそのまま紙・FAX の申込書になる。デジタルと紙で
  二重の様式を持たない

この便利さは、xlsx が事実上の共通言語(誰の手元でも開ける・書ける・印刷
できる)であることに由来する。VBA を分離した(次節)うえでの神Excel は、
仮の回避策ではなく本格の部品である(§0)。実装例は seminar-kit(§13)。

### VBA を使わない設計(セキュリティの要)

様式の自動化は VBA ではなく、役割を2つに分離して実現する。これにより
「文書を開くこと自体がコードの実行になる」という VBA の攻撃経路を無くす。

- **外部 Python(処理)**: 本文テキスト・様式の読み取り、検証、登録を行う独立プログラム。
  文書の外にあり、git 管理・監査可能。ファイルを開いても起動しない
- **内部 JavaScript(補助)**: OnlyOffice のマクロ(サンドボックス内)。送信用
  テキストの組み立て等の入力補助のみで、システム権限を持たない。無くても
  数式・添付のフォールバックで成立する(OnlyOffice 未使用でも申込できる)
- データ(様式)と実行コードが同居しない。受信側の処理は常に外部 Python が
  行い、様式は保護とデータだけを持つ

手で書くなら、VBA も Python・JS も簡単ではない——書ける一部の人に属人化し
(「開かずのマクロ」)、しかも危険だった。この手書きの敷居は AI が下げ、
属人化はむしろ解消する(§7)。

### 派生生成の順序

様式プロファイル(AsciiDoc)から各派生物を生成する順序は固定する:
**DDL → Pydantic → 記入用テキスト → xlsx**。制約(SQL 語彙)が最初に
確定し、以降の派生はすべてその制約に従属する——検証がどこかの層だけ
甘くなる、という食い違いを構造的に防ぐ(前掲「検証がずれない」の
プロファイル版)。

### 回答原稿AI(問い合わせ対応の拡張。アプリ層の opt-in モジュール)

問い合わせ対応は「非構造の文章が届く → 分類する → 過去の類似案件を探す →
回答を書く → 送る」の反復である。過去の回答は組織に蓄積されているのに毎回
ゼロから人が書き、担当者の退職で回答ノウハウも消える。この構造に、
ローカルAI を**原稿書き(書記)**として足せる:

- **分類**: 受信した問い合わせを小型ローカルモデルが分類し、該当する公開
  様式(dist/forms/)への案内文を下書きする。案内は様式への**リンク**で行い、
  ファイルは添付しない(添付なし主経路のまま)
- **回答原稿(RAG)**: 過去の回答・FAQ・規程文書から pgvector で類似を検索し、
  Command A+(§7)が**引用付きの回答原稿**を生成する。原稿には必ず引用元
  (どの過去回答・規程に基づくか)を添え、引用が空なら「根拠なし」の警告を付ける
- **決裁(全件)**: 担当者は原稿+引用元を見て「そのまま送信/修正して送信/
  破棄して自分で書く」。送信の主体は常に人間で、自動送信という概念が存在しない。
  「AI が誤った回答を送った」が構造的に起きず、差し止めタイマー等の複雑な
  仕掛けも不要で、責任の所在が明快
- **蓄積**: 送信された最終回答が FAQ として蓄積され、次回以降の原稿の材料に
  なる(使うほど自組織専用に精度が育つ)。担当者の修正差分は原稿品質の指標。
  属人化していた回答ノウハウの形式知化が同時に進む

段階導入とする: (1) **蓄積**——分類のみ稼働し、回答は全件人が書く(それが
FAQ データになる)。(2) **原稿**——蓄積が進んだら RAG 原稿を開始、全件承認。
(3) **成熟**——無修正で送信される割合が上がれば、作業は実質「確認と送信」に
近づく。それでも送信の主体は人間のまま。

**入口はメール(既定)**。本モジュールは §5 のメール受付(IMAP)に対して
働くため、公開層に動的なフォームを足さなくても成立する(公開層静的の原則を
維持)。一次受付の Web フォーム(メールアドレス+250字以内の「ご用件」だけを
受け、分類結果で該当様式へ案内する)を置くこともできるが、これは任意の追加
入口であり、導入する場合は §11 の対策一式(メールボム対策・成功偽装応答)を
必須とする。名前・会社名を入口で求めない——攻撃側が容易に用意できる情報は
防御に寄与せず、正規利用者の障壁になるだけ。詳細は様式の記入(=依頼の意思が
ある段階)で初めて収集する。用件は選択式にせず自由記述とする——選択式は
機械的に列挙できて防御に無力なうえ、自由記述の方が分類の手がかりも多い。

送信テキストの形式は §5 のまま(`項目: 値` の寛容パーサ)。YAML で届いても
読める(まず YAML として試し、失敗したら寛容パーサに落とす)が、利用者に
厳密な書式は要求しない。注文明細のような繰り返し項目は「1行1明細」の行形式で
表す(寛容パーサの範囲。スキーマが唯一の定義である点は様式方式と同じ)。

Copilot 等の汎用AIアシスタント(§0)との違い: 接地が自組織の文書に限定され
引用が構造化されて返る/人間が全件承認し責任が明快/データが建物の外に
出ない(§7)/コストは固定費のみで使うほど1回あたりが下がる/蓄積が
そのまま自組織の FAQ データベースとして手元に残る。

## 6. メール

- 受付・自動返信・振り分けを担う。段階導入とする
  - 段階1: 既存のメール環境(SMTP/IMAP)を利用
  - 段階2: 自営メールサーバー Stalwart(SMTP/IMAP/JMAP・スパム対策・DKIM)へ移行。
    リスクの低い受信から自営化し、送信は到達性の検証が済んでから切り替える
- 送信は**自社回線の固定IP**から直接行う。自社の固定IPは、クラウドの共有IP帯
  (スパム送信に使われ評判が汚れがち)と違って送信履歴が汚れておらず、
  レピュテーションを自ら管理できる
- 送信切替前の確認項目: SPF/DKIM/DMARC・逆引き(PTR)の DNS 整備、
  主要プロバイダ宛の到達確認
- 送信処理・受信処理はそれぞれ1モジュールに集約する

## 7. ローカルAI(利用範囲を限定する)

AI は用途(コード/文書)と内容の機微(公開/内部秘)で使い分ける
(docs/05 と同じ二層)。基本はクラウド(Claude)、局所・内部秘だけローカル。

- コード: 基本は **Claude**(キット・生成器・パーサ・様式マクロの基盤・
  移行スクリプト=非機微)。細かい調整・現場の小さな自動化・内部データに
  触る分は **North Mini Code**(ローカル・VBA の代替)
- 文書: 基本は **Claude**(公開情報の変換・下書き)。内部秘・個人情報を
  含む分は **Command A+**(ローカル)
- 原則: 公開・非機微 → クラウド(Claude)/ 内部秘・個人情報・現場の局所
  → ローカルで建物の外に出さない

### なぜ機微を API に出せないか(借りられる境界がない)

この線引きは慎重さの好みではなく、構造から来る。

- Web のクラウド(IaaS)は**計算機の境界を借りる**契約である。ハイパー
  バイザの隔離の内側は自分のもの(自分の OS・自分の鍵・自分の認証)で、
  内部秘の Web システムが金融・行政でもクラウドで動くのはこのため
- AI の API(SaaS)は**データを相手のアプリケーションに渡す**契約である。
  プロンプトは相手のサービスの中で復号され、相手の GPU 上で処理される。
  経路の暗号化は計算の瞬間を守らず、契約上の保証(学習に使わない・保存
  期間の限定)は物理的な管理の代わりにならない。借りられる境界が存在しない
- 原因は対称的で、**フロンティアモデルの重みは提供側の機密**だからである。
  重みが渡されない以上、計算は必ず相手の境界の中で起きる。逆にオープン
  ウェイト(Command A+・North Mini Code)は重みがこちらに来るので、
  **計算ごと自分の境界に持ち込める**——ローカルAI が機微を扱える唯一の
  AI の形である理由
- ゆえに API への防御は「何を出すか」の線引きだけであり、それが上記の
  原則(公開・非機微のみ)である。Claude を運用責任者の1席に限る(§0)
  ことは、この線引きを守る人間を一人にする、という機密面の意味も持つ

まとめると統一原理は一つである: **開放とは、計算を自分の境界に取り戻す
こと**。OSS はアプリケーションを、オープンウェイトはモデルを、自分の
境界に持ち込む。外に残すのは、境界を借りられる仕事(公開 Web)と、
公開してよい情報しか触れない1席(Claude)だけ。境界の持ち方は三段あり
——共有 SaaS(境界なし)/ソブリン提供(交渉力で作らせる専用境界。
政府クラウド等、国家規模の契約力が要る)/自営(自分で持つ)——
**OSS 自営は、交渉力のない中小組織にとってのソブリンクラウドの代替**である。

### オープンウェイトは OSS と違い検査できない

§0 の部品表が「OSS」と「オープンウェイト」を書き分けているのは、
検査可能性が本質的に違うからである。

- OSS の透明性は「読めば挙動が分かる」に立脚する。ソースはレビューでき、
  ビルドし直せば配布物との一致も検証できる
- オープンウェイトで手に入るのは読めない数値の塊である。訓練データと
  訓練過程は公開されず、再ビルドも差分レビューもできず、振る舞いの試験は
  「特定の引き金で発火する仕込み」の不在を証明できない
- 検証で信頼を置き換えられない以上、残るのは**訓練者への信頼**だけである。
  ゆえにモデルの選定は設計事項であり(本書が名指しで決める)、利用者の
  日々の選択肢にしない。無保証(AS IS)のライセンスを引き受けられるのは、
  自分の境界で・自分のために・緩和策込みで使う場合に限る
- 設計の答えは**監査対象の移動**である: モデルは検査できなくても、その
  出力(Python・JS・Markdown)は読める・差分が取れる・凍結できる人工物
  である。人が確認して git に凍結し、AI をランタイムに置かない(§7 冒頭)
  ——検査可能性をモデルから成果物へ移す
- 副次的な利点: 重みは固定できるので挙動が黙って変わらない。検査可能性の
  序列は OSS > オープンウェイト(固定可)> API(固定すら不可)

設置場所は限定する:

- 置かない: 公開物(静的HTML)、および申込→登録の決定的な処理経路
  (決定性・再現性の確保、受信メール経由のプロンプトインジェクション対策)
- 使ってよい(開発時): コードの基盤(パーサ・生成器・様式マクロ・移行
  スクリプト)は Claude、細かい調整・局所の自動化は North Mini Code。記事の
  HTML→Markdown 変換・文章の下書きは公開=Claude / 内部秘=Command A+。
  **出力(Python・JS・.md)は人が確認して git に凍結**し、公開物・処理経路には
  AI を残さない
- 使ってよい(運用): 記事・案内文の下書き、アンケート要約・返信文の下書き
  (Command A+。結果は人が確認してから用いる)

§5 の回答原稿AI はこの線を系統化したものである。AI を置くのは**提案経路
(原稿)**であり、決定経路(登録・管理番号の発行・送信)には置かない。原稿は
全件が人間の承認を通り、承認された回答が FAQ として蓄積される——「監査対象を
モデルから出力へ移す」原則(前節)の運用版である。受信メール経由のプロンプト
インジェクションが汚せるのは「人が読む原稿」だけで、決定経路には届かない。
機微の線引きも同じ: 受信した問い合わせ文は外部者の個人情報を含みうるため、
分類・原稿生成とも外部 API に出さず、ローカル(分類=小型モデル、原稿=
Command A+)で行う。LLM 呼び出しは1モジュールに集約し、モデルの差し替えは
その1箇所で済ませる(汎用の抽象層は作らない。§13 の原則と同じ)。

### AI の持ち場(境界の明示)

様式プロファイル(§5)まわりで AI が担う範囲は次の3つに限る。いずれも
「提案まで」であり、決定経路(DB 書き込み・登録)には置かない。

1. **pending の一次処理**: 修復不能な入力の解釈案をローカル LLM
   (Command A+)が付けて pending に置く。AI は DB に直接書かない
2. **様式定義の生成**: 既存の紙様式・Word 等から SQL 語彙付き AsciiDoc
   プロファイルの下書きを生成する。出力の正しさは制約とテストが機械検証する
3. **表示層の解釈**: 自由記述の説明文を読んで xlsx レイアウト・記入
   テキスト整形に反映する

LLM はオプショナルであり、無ければルールベースにフォールバックする
(mdit-py-cjk-friendly と同じ流儀。classify も同様)。

### North Mini Code の位置づけ(VBA の代替=誰でも安全な自動化を作れる)

VBA を外部Python+内部JavaScript に分ける設計(§5)は安全だが、手書きの
ハードルは高い。VBA も同じく簡単ではなく、書ける一部の人に属人化していた
(「開かずのマクロ」)。基盤のコード(パーサ・様式マクロ)は Claude が書き、
運用で必要になる現場の細かい調整・小さな自動化・内部データに触るコードは
North Mini Code が支える。要点は「AI が肩代わりする」ではなく、**専門家で
なくても、現場の担当が自分で安全な自動化を作れるようになる**こと。開発者採用
やベンダー委託(恒久コスト・依存)にも、特定の書ける人にも縛られない。

- 安全: ローカル実行。コードや実データを建物の外に出さない
- 低コスト: オープンウェイト(Apache)。席数課金・月額なし(電気代のみ)
- 裾野が広がる: かつて VBA で非プログラマが自作していた自動化を、いまは
  誰でも安全な Python・JS で作れる(「書ける人の異動で死ぬ開かずのマクロ」も、
  AI が支えることで属人化しない)
- あくまで**開発時の生成**。出力(Python・JS)は人が確認して git に凍結し、
  ランタイム・処理経路には AI を残さない(§5 の外部Python は AI ではなく
  凍結された通常のコード)
- かつて現場が VBA で作っていた自動化を、ローカルAI が生成する安全な
  Python/JS に置き換える、という位置づけ

### Command A+ の位置づけ(Markdown化・文章作成の鍵)

内容を Markdown で git 管理する本格の形(§3)も、同じ論理で AI が鍵になる。
CMS の WYSIWYG クリック編集に慣れた担当者には、Markdown を書くことも、旧CMS
の HTML を大量に Markdown へ移すことも、そのままでは難しい。ここを外注すれば
恒久コストと依存が戻る。Command A+ がその変換役・下書き役を担う。

- 安全: ローカル実行。内部秘・個人情報を含む文書も建物の外に出さない
  (公開情報はクラウド〔Claude 等〕で足りる=機微で使い分け。上記のとおり)
- 低コスト: オープンウェイト。席数課金・月額なし
- あくまで**開発・運用支援**。出力(.md)は人が確認して git に凍結し、公開物
  (静的HTML)や処理経路に AI を残さない(生成物はただの Markdown)
- 位置づけ: CMS の編集UI(の下書き・変換の手間)を、ローカルAI による
  Markdown 生成・整形に置き換える

**まとめ**: 本格の設計(VBA-free のフォーム、Markdown の内容)は手作業では
authoring の敷居が上がる。その敷居を AI が下げ、**専門家でなくても誰でも
作れるようになる**。機関は本格の形を自力で・安く・安全に持てる。

### 表計算を薄くする(データ=宣言的 / 計算=Python / 体裁=テンプレート)

Excel の多重関数リンクは、データ・ロジック・体裁をセルの中で絡ませる——壊れ
やすく・不透明・版管理もテストもできない。重い集計はピボットやアドインに
閉じ込められ、席数ライセンスと属人化を生む。**この分離を邪魔しているのが
Excel(表計算を"アプリ本体"にする使い方)**。移行後は表計算を薄くする:

- データ → **YAML/JSON**(宣言的・git 差分・行移動で壊れない。§3 の構成データ)
- 計算 → **Python**(監査・テスト可能・限界費用ゼロ。ピボット/アドインの代替)
- 体裁 → **テンプレート**(生成物として分離)
- 表計算(OnlyOffice)は「**記入・表示**」に軽くする。ロジックの置き場にしない

橋渡しはマクロ: OnlyOffice の JS マクロが記入済みシートを構造化 JSON/YAML に
変換する。**マクロは AI(North Mini Code)が開発時に生成し、runtime は決定的な
素の JS**(ランタイムに AI を置かない)。流れ: 記入(OnlyOffice)→ マクロ
(AI製・決定的)→ JSON/YAML → Python・テンプレート。CMS/Excel が一手に抱えて
脆くしていたものを、それぞれ得意な場所に返す。

## 8. サーバー構成・稼働

| 区分 | 稼働先 | 内容 |
|---|---|---|
| 公開サイト | Cloudflare Pages | 静的HTML。自サーバーに公開Webを置かない |
| 認証(アプリ層) | 機関内 | PocketBase。基本的な認証。職員・会員を役割で分離(モジュール利用時) |
| git 共同編集 | 機関内 | Forgejo(認証内蔵)。記事(.md)・データ(.yaml)の編集・版管理 |
| 編集クライアント | 担当者端末 | Zed(build/確認/配信をタスク化)・ブラウザ(OnlyOffice/Forgejo) |
| 文書・様式編集 | 機関内 | OnlyOffice Docs(編集エンジン)+ 自作文書ホスト(§2。認証は PocketBase) |
| メール | 機関内(段階2で Stalwart) | 送受信・受付・自動返信 |
| データ | 機関内(アプリ層) | PostgreSQL(モジュール利用時) |
| ロジック/API | 機関内(アプリ層) | FastAPI。受信処理・モジュール(モジュール利用時) |
| ローカルAI | 機関内 | North Mini Code 等。開発・運用支援(§7) |

- 公開層(Cloudflare)は常時稼働。編集・運用層/アプリ層は機関内の自営
- 機関内のサーバーはインターネットに直接面させない。公開が要る接点のみ限定
- 最小構成では編集・運用層とアプリ層を簡素にできる(規模で選ぶ)
- 移行完了後、旧CMS・公開Webサーバーは退役する
- 公開面のパッチ運用(CMS・PHP・Webサーバー)が不要になる。ただし機関内
  スタック(Forgejo・OnlyOffice・Stalwart・PocketBase・PostgreSQL 等)の更新は
  残る——攻撃面が公開Webから消える代わりに、運用は機関内に引き受ける(§9)

### 配置の理由(Web は外へ、AI は中へ)

上の表の配置は、仕事の性質から決まっている。

- **Web の仕事は小さく・短く・状態を持たない**(リクエスト1件は数ミリ秒・
  どのコピーが応えても同じ)。だから仮想化・複製・キャッシュが効き、
  クラウドに向く。静的配信はその極限で、公開層を Cloudflare に出すのは
  この性質を使い切る配置である。内部秘の Web もクラウドは選択肢になり得る
  (境界を借りられる。§7)——本設計がアプリ層を機関内に置くのは機密上の
  必然ではなく、**この規模なら手元が安く・単純で・依存が減る**という選択
  (既存サーバーの流用・従量課金なし・「建物の中にある」の一言で検証が済む)
- **AI の仕事は巨大・常駐・専有**。モデルの読み込みだけで数分かかり
  (100GB 級の重みを GPU メモリへ転送)、GPU メモリは複数の利用者で薄く
  共有できない。「必要なときに立ち上げ、済んだら畳む」という仮想化の
  弾力性が原理的に使えず、クラウドの旨味が消える。だから買い切った実機に
  載せたままにする(工作機械に近い運用。電気代だけで常時温まっている)
- 費用も同じ結論を出す: **API は常駐コストを大勢で割るから最も安い**
  (ただし機微は出せない。§7)。**自前の実機は買い切り+電気代**。
  **クラウドの仮想環境で AI を自営するのは、常駐費とマージンを独りで
  負担する最も高い形**で、費用・機密の両軸で劣るため採らない
- まとめ: 仮想化が効く仕事(Web)は外へ、常駐する仕事(AI)は実機へ、
  外に出すのは公開してよい情報だけ(§7)

## 9. 運用手順

- 日常更新: content/ の .md を編集 → build → 確認 → 配信
- 共同編集: Forgejo(機関内 git)で複数担当が編集・レビュー・版管理。
  編集は Zed(build/確認/配信をタスク化)。文書・様式は OnlyOffice Docs でも編集可
- 版管理: git(誰がいつ何を変えたかを履歴で追える)
- サーバー更新: 機関内スタックは各サービスをコンテナで持ち、定期(月次目安)に
  一括更新する。更新は一つのスクリプト(またはタスク)にまとめ、手順書に含める。
  機関内サーバーはインターネットに直接面していない(§8)ため、公開Webの
  パッチ運用のような緊急性はない
- 監視: **公開Webとメールだけ**簡易な監視を置く。この二つは止まっても
  「外から来るものが来なくなるだけ」で内側からは気づきにくいため——公開URLの
  外形確認と、自分宛て送受信の疎通確認を定期実行して失敗を通知する、小さな
  スクリプトで足りる。編集層・アプリ層(Forgejo 等)は止まれば使う人がすぐ
  気づくので、専用監視は置かない
- バックアップ: ソース・記事は git(Forgejo のリポジトリを外部へ複製)。
  PostgreSQL はダンプ、メールストア・文書ホストの実体ファイルと権限ストアは
  ファイル複製で、いずれも機関内の別筐体へ定期取得する。受信原本・受付データも
  同様に機関内で保全
- 引き継ぎ: 「編集→生成→配信」の手順書を成果物に含める。担当者が
  一巡できることを到達目標とする

## 10. 移行手順

**順序の要:**

- **まず共同編集の基盤(Forgejo+Zed)を立てる**——CMS の核=複数人編集の
  置き換え。内容はここに載せて複数担当で編集する(§1)
- **内容の入口は「公開分を Claude で Markdown 化」**。公開情報は外に出る内容
  なのでクラウド(Claude)で変換してよい。効果確認・デモは1人でも小さく始められる
- 内部秘の変換(Command A+)・問い合わせ様式・アプリ層モジュールは、
  必要になった段階で足す(§7 の機微による使い分け・§13)

以下は全体の順序:

1. 共同編集の基盤(Forgejo+Zed)を立てる。内容データは機関側で用意する
   (CMSエクスポート・既存原稿等)。
   データがあればクロールは行わない。クロールは了解を得たうえでの最後の手段
2. 記事=MD / 構成=データ に変換する(機械変換+人の仕上げ)。HTML→Markdown の
   下書き・抽出/変換スクリプトには AI を使う(機微による使い分けと git 凍結の
   原則は §7)
3. テンプレートで生成し、並行公開して確認する
4. 切替後、既存URLはリダイレクトで維持する
5. 旧CMSを退役する(保守・ライセンス・脆弱性対応の負担が解消する)

## 11. セキュリティ

- 攻撃面: 公開面から動的CMS・PHP・フォーム処理を除去する
- 配信: Cloudflare が TLS・WAF・DDoS 対策・セキュリティヘッダ(HSTS/CSP 等)を
  担う。HTTP/2/3 も自動
- 個人情報: 公開サイト・外部クラウドに置かない。問い合わせ内容はメールで
  機関内に届き、機関内で保全する
- 真正性: 送信者アドレスで本人性を確認する
- **配布様式の真正性**: dist/forms/ に載せる様式の SHA-256 を併記する
  (build が dist/forms/sha256.txt を出力。public/ のコピーより後に書き、
  ハッシュ表と実物の食い違いを構造的に防ぐ)。様式には site.yaml 由来の
  識別子(`受付:` 行。隠しセル→送信テキスト・マクロ・記入テキストに
  自動で流れる)を埋め、受付側(parse)で照合する——識別子が有って
  違う様式は**返信せず** pending へ(差出人は詐称されうるため、返信すると
  バックスキャッター発生源になる)。無い場合は寛容に受ける(手書きを
  弾かない)
- **様式指紋**: 項目構成(ラベル+制約の並び)のハッシュ(`様式指紋:` 行)を
  様式に埋め、受付側で照合する。FORM_VER(人が上げるプロトコル版)と違い
  定義から機械的に導出されるため、項目の挿入・入替で位置がずれた旧配布
  xlsx を確実に検出できる——こちらは正規利用者の事故なので、最新様式への
  案内を自動返信する
- pending の添付を人が開くときはマクロ無効環境(既存規約の維持)
- 様式の自動化に VBA を使わない: 外部Python(処理)+内部JavaScript(サンドボックス)
  に分離し、データと特権実行コードを同居させない(§5)。「開く=実行」の攻撃面がない
- 公開 Web フォームを置く場合(§5 回答原稿AI の任意入口)は、メール送信の
  踏み台化への対策を必須とする: 同一宛先への送信頻度制限(例: 1分1回・
  1日3回)+同一IPの頻度制限、**制限超過でも成功と同じ応答を返す**(何回で
  弾かれるかを攻撃側に教えない)。CAPTCHA 単体には頼らない——AI による突破が
  実用化しており、抜かれる前提で、性質の異なる層(頻度・内容の意味的妥当性・
  AI 異常検知)を重ね、どの層が抜かれても後続で吸収する。攻撃されるほど
  請求が増える従量課金型のボット対策(reCAPTCHA Enterprise 等)は用いない。
  WAF・IPレピュテーション(Coraza・CrowdSec 等の OSS)は本体側の対策を
  固めた後の次フェーズ
- 責任の整理(納品時): OSS 構成は無保証(AS IS)であり、保守契約で責任範囲を
  明記する。サイバー保険が補償するのは単発事故(ランサムウェア・個別の漏洩)で、
  クラウド障害・共有ベンダー経由の同時多発のようなシステミックな事象は契約上
  除外される。**最小依存・自営は、保険で救済されないこの相関リスクの土台
  (共有依存)自体を減らす**実効的な低減策である(§7 の三段の境界と同根)

## 12. リポジトリ・技術構成

```
aiseed-migration-kit/
  src/amig/          cli / site(設定)/ ingest(取り込み)/
                       classify(分類)/ convert(MD化)/ build(生成)/
                       publish(cf-publish 呼び出し)/ rules(既定の抽出ルール)/
                       decision(決裁部品: 属性解釈・インデックスSQL・凍結記録。§14)
    inquiry/           profile(様式プロファイル解釈)/ spec(SQL 語彙・修復)/
```

- cf-publish の同梱(vendor/)は PyPI 公開までの暫定。公開後は通常の
  依存(pip)に切り替える
- ライセンス: 本キット自体は **AGPL-3.0**。構成部品は OSS(AGPL/Apache 系)。
  CMS のような継続利用料は生じない。AGPL の理由と SME 社内利用への影響は
  README に記載する
- 中身はすべてテキストと標準ツールで、監査・引き継ぎが可能

## 13. アプリ層とモジュール(業務システム・決済・会員機能)

アプリ層は、認証・DB・メールを共有基盤とし、その上にモジュールを載せる。
公開層(静的)は変えない。機関内で稼働し、インターネットに直接面させない。

### 共有基盤(モジュールの土台)

- **認証(PocketBase)= アプリ層の必須コンポーネント**。基本的な認証のみを
  担い、職員(業務システム操作)と事業者会員(会員ポータル)を**役割(role)で
  権限分離**する。アプリ層のモジュールは独自にログインを作らず、これを使う。
  編集層(Forgejo)は内蔵の認証を使い、この基盤には統合しない(§2)
- **データ(PostgreSQL)**: モジュールはスキーマを分けて同一DBに同居できる
- **メール**: 通知・受付・自動返信を共有(§6)
- 個人情報・認証情報はこの層(機関内)に閉じ、公開層・外部クラウドに出さない

### モジュール

| モジュール | 例 | 設計 |
|---|---|---|
| 業務システム | 研修・セミナー管理(申込→名簿→台帳)、依頼試験・設備予約の受付管理 | 事務局アプリ(FastAPI/DB直結、UI は Flet 等)。認証は共有基盤。**seminar-kit がこの形の実装例**(職員認証に PocketBase を使用) |
| 会員(事業者ポータル) | 事業者のログイン・申請状況照会・継続利用 | 認証は共有基盤(会員 role)。公開層からはログイン画面へ誘導するだけ。個人データはアプリ層に閉じる |
| 文書ホスト | 社内文書の保管・ブラウザ編集(OnlyOffice Docs 統合) | 自作の最小ホスト(§2)。実体=ファイルシステム、権限=構造化ストア、認証は共有基盤 |
| 決済連携 | 受講料・手数料の収納 | カード情報は自組織で保持せず**外部の決済事業者に委譲**(PCI DSS 範囲を持ち込まない)。可能なら請求書・公金収納(証紙代替の指定納付等)に寄せる。アプリ層は収納状況のみ保持 |
| 回答原稿AI | 問い合わせの分類・引用付き回答原稿・FAQ蓄積(§5) | PostgreSQL+pgvector、承認画面は事務局アプリ(Flet 等)。LLM はローカルで、決定経路に置かず全件人間承認(§7)。カスタマーサポート・見積一次回答・社内規程照会・クレーム一次対応など「分類→過去の蓄積→原稿→人が決裁」の形の業務に共通で使える |

### 原則

- **公開層は静的・匿名・個人情報なし**を保つ。動的性・認証・個人情報は
  すべてアプリ層(機関内)に閉じ込める
- **認証はアプリ層の基盤**であり、モジュールごとに作らず共有する
- モジュールの追加はアプリ層への追加であって、公開層(§2〜§5・§8〜§12 の
  静的部分)の設計は変わらない
- 共有基盤は「既知の要件のための土台」として持つ。個々のモジュールの内部に
  憶測の拡張点・汎用設定は作らない(必要になったモジュールを単純に書く)

## 14. 決裁部品(文書管理)

決裁項目(承認者・状態・決裁日等)は**文書内の AsciiDoc 属性が正**
(`:approver:` `:status:` `:decided-on:` 等)。PostgreSQL は全文書の属性を
集約した検索・一覧用インデックス(派生物)。DB は rebuild で再生成できる
(§0「データの置き場」の判断=git・記録=PostgreSQL の対応と同型)。

- 決裁フローの実体は Forgejo のプルリクエスト: 起案=ブランチ / 審査=行
  コメント・PR レビュー / 決裁=マージ(署名付きコミット)/ 施行=build
  して配信
- コメントは AsciiDoc の `//` 行コメント・`////` ブロックコメント
  (出力に現れない)。ただし履歴には残るため、議論は PR 側・文書内コメントは
  事実的注記のみ、を規約化する
- 人事・個人情報系の決裁は最初から別リポジトリに分離する(アクセス制御は
  リポジトリ粒度で設計)
- 交付物(請求書・修了証・決定通知等)は PDF/A で凍結し、生成元
  (AsciiDoc + コミットハッシュ)と交付物ハッシュを併せて保存する
  (§0「データの置き場」の交付の凍結)。凍結記録は上書きしない(convert と
  同型の保護)。生成元に未コミットの変更がある間は凍結を拒否する——
  記録するコミットと実際の生成元が食い違う「誤証拠」を作らない
  (決裁=マージ済みの文書から交付する)

様式プロファイル(§5)と同じ発想の適用先である——「文書内の属性が正、
DB は再構築可能な派生物」を、申込データだけでなく決裁文書一般に広げる。

## 15. 同一基盤の実証(定義差し替えのみで成立すること)

sites/seminar・sites/mfg に、既存の2キット(seminar-kit・mfg-kit)の
受付様式を**定義ファイル(site.yaml+様式プロファイル)だけ**で再現して
いる。コードは1行もない——tests/test_showcase.py が「同一基盤の様式定義
差し替えのみで成立する」ことをテストで示す。

- **seminar**(研修・セミナー管理): 現行 WordPress 埋め込みフォームの写像
  (企業情報+受講者最大3名+参加場所 会場/ZOOM/サテライト)。受講者の
  繰り返しは現行の上限どおり固定3組。フリガナのカナ制約が「ひらがな→
  カタカナの決定的修復/漢字は修復不能で pending」の実演を兼ねる
- **mfg**(製造業の見積・受注): 見積依頼。明細(製品×数量×個別仕様)の
  繰り返しは「1行1明細」の行形式(§5)で text 制約に収め、子テーブル
  (quote_items)への展開は受付後のアプリ側の仕事とする。和暦の希望納期が
  date 制約への修復の実演を兼ねる

可変長の明細を様式プロファイルの語彙として持つか(子テーブル 1:N の
宣言)は、固定組・行形式で足りなくなった時点で判断する(憶測の拡張は
作らない。§13 の原則)。

新しい業務への適用も同じ形を取る: 個別実装を作らず、様式プロファイルを
1枚書く。法令・業務の要件は制約(SQL 語彙)として様式に焼き、表示・案内は
本文の自由記述に書く——それで足りない時に初めて基盤側の拡張を検討する。
